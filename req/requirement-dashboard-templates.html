req-008

Send automated status-change notifications to customers

v1.0 Approved
Specification

The system SHALL send an automated notification to the customer when their order status changes to "Shipped" or "Delivered". The notification SHALL be delivered via email. The email SHALL include: order reference, new status, estimated delivery date (if available), and a link to the order detail page. Notifications SHALL be sent within 10 minutes of the status change event being received.

Acceptance Criteria
  • ac-017 Given a "Shipped" status event is received, when 10 minutes have elapsed, then the customer's registered email address has received a notification email containing the order reference and estimated delivery date.
  • ac-018 Given a "Delivered" status event is received, when 10 minutes have elapsed, then the customer receives an email confirming delivery and containing a link to the order detail page.
  • ac-019 Given the email service is unavailable when a status event arrives, when the service recovers, then the notification is sent via a retry queue within 1 hour of the original event.
Assumptions
  • asm-008 The customer's preferred notification language defaults to the language set on their profile. If not set, English is used.
Open Questions
  • oq-008 Should customers be able to opt out of status notifications individually, or is notification mandatory for all confirmed orders?
Risks
HIGH risk-008 Sending emails without an opt-out mechanism may breach email marketing regulations in some jurisdictions (e.g. CAN-SPAM, CASL). Mitigation: add an unsubscribe link and honour opt-outs before v1.0 release.

req-002

Detect and reject duplicate email addresses at registration

v1.0 Approved
Specification

The system SHALL reject a registration attempt if the submitted email address already exists on any active or suspended customer record. The user SHALL be shown a message directing them to the login or account-recovery flow. The check SHALL be case-insensitive.

Acceptance Criteria
  • ac-003 Given an email that already exists on an active record, when a new registration is submitted, then the system rejects the attempt with status 409 and returns a human-readable message.
  • ac-004 Given an email that differs only in casing (e.g. User@example.com vs user@example.com), when a registration is submitted, then the system treats it as a duplicate and rejects the attempt.
Assumptions
  • asm-002 Deleted/archived customer records are excluded from the duplicate check unless the business rules are revised.
Open Questions
  • oq-002 Should the system reveal whether the email belongs to a suspended account, or give a generic "email already in use" message for security reasons?
Risks
HIGH risk-002 Revealing account existence through the error message could aid account enumeration attacks. Mitigation: use a generic message and add oq-002 to the security review.

req-004

Audit log for all customer profile changes

v1.0 Approved
Specification

The system SHALL maintain an immutable audit log for every change made to a customer profile record. Each log entry SHALL include: the field changed, the previous value, the new value, the actor (customer or staff), the actor's user ID, and the UTC timestamp. The log SHALL be retained for a minimum of 3 years.

Measurable Thresholds

Compliance Context
  • Retention: 3 years minimum, write-only from application layer
  • GDPR tension: right-to-erasure vs. immutable log (risk-004)
  • Lawful basis: pending DPO confirmation
Acceptance Criteria
  • ac-007 Given a customer updates their phone number, when the change is saved, then an audit log entry is created within 1 second containing the old value, new value, customer user ID, and UTC timestamp.
  • ac-008 Given an audit log entry is created, when a DELETE or UPDATE is attempted on that entry via any application path, then the operation is rejected.
  • ac-009 Given a log entry is 3 years old, when a retention query runs, then the entry is not purged and remains queryable.
Assumptions
  • asm-004 The audit log is write-only from the application layer. No update or delete operations are permitted on log entries.
Open Questions
  • oq-004 Is the audit log exposed to customers (self-service view of "your recent changes") or only accessible to internal support staff?
Risks
HIGH risk-004 Storing previous field values may include sensitive personal data (e.g. old phone numbers). GDPR right-to-erasure requests may conflict with the immutability requirement. Mitigation: consult DPO before finalising retention policy.

req-006

System validates stock availability before order confirmation

v1.0 Approved
Specification

Before confirming an order, the system SHALL check stock availability for each line item against the inventory service. If any line item is out of stock, the order SHALL NOT be confirmed and the customer SHALL be shown which items are unavailable. The stock check SHALL be performed in real-time at confirmation, not at the time the item is added to the draft.

Acceptance Criteria
  • ac-012 Given a customer confirms an order where all line items are in stock, when the confirmation is submitted, then the order status changes to "confirmed" and an order reference is returned.
  • ac-013 Given a customer confirms an order where at least one line item is out of stock, when the confirmation is submitted, then the order is not confirmed and the response lists the out-of-stock items by name and SKU.
  • ac-014 Given the inventory service is unavailable, when the confirmation is submitted, then the order is not confirmed and the customer is shown a service-unavailable message with an instruction to retry.
Assumptions
  • asm-006 The inventory service provides a synchronous REST endpoint for stock queries and responds within 2 seconds under normal load.
Open Questions
  • oq-006 Should the system allow partial order confirmation (confirm available items, hold unavailable items) or require all items to be available before confirming?
Risks
HIGH risk-006 A race condition may allow two customers to confirm orders for the last unit of stock simultaneously. Mitigation: inventory service must use optimistic locking or reservation tokens.

/req - Requirements Management Skill

Coordinates specialist agents to turn messy business input into structured requirements, MVP scope, workflow maps, and quality constraints.

Entry point - how to invoke
$ /req Draft requirements for a CRM case search
Produces a requirements draft with acceptance criteria
$ /req Review these requirements and make them testable
Reviews and improves measurability of existing requirements
$ /req Cut this order flow to MVP
Reduces scope to a minimum viable product
$ /req Map the workflow before writing requirements
Clarifies current and target process before drafting
$ /req Identify missing NFR for a customer portal with personal data
Exposes security, privacy, and compliance gaps
$ /req Analyze integration requirements for Stripe and ERP
Maps system-to-system behavior and failure expectations
$ /req Compare two options for authentication
Structured trade-off analysis with recommendation
$ /req Prepare meeting about order management
Agenda, blocking questions, and decision points

Specialist agents - triggers and output
requirements-analyst
Turns business needs into structured functional requirements with acceptance criteria
Always active - also triggered by
draft requirements review requirements user stories acceptance criteria clarification needed meeting prep
Output
REQ-[ID] statements, AC-[ID].[N] acceptance criteria, assumptions, open questions, specialist triggers
process-analyst
Maps current and target workflow so requirements are grounded in how work actually happens
Triggered by
workflow changes approvals handoffs roles manual steps exception paths operating process
Output
Current process, target process, actors, responsibilities, handoffs, exceptions, process risks
scope-strategist
Reduces complex or broad requirement sets to a buildable MVP without losing core user value
Triggered by
large scope many functional areas cross-system spread "full solution" "everything" "complete" cut to MVP
Output
Minimum value statement, MVP scope, excluded items and consequences, later scope, MVP requirements draft
integration-analyst
Identifies missing behavioral requirements for system-to-system interactions
Triggered by
API calls data flows events queues webhooks imports / exports system dependencies
Output
Integration points, missing behavioral requirements, failure expectations, ownership, REQ-INT statements
nfr-analyst
Identifies missing quality constraints - security, privacy, compliance, availability, and operations
Triggered by
security privacy / GDPR availability monitoring compliance SLA production readiness
Output
NFR-[CATEGORY]-[ID] proposals, candidate thresholds, done-means checklist, GDPR analysis when relevant
testability-analyst
Turns vague or subjective requirements into verifiable ones without changing business intent
Triggered by
vague language subjective terms unclear criteria non-verifiable make testable missing thresholds
Output
Vagueness classification (High/Medium/Low), measurable interpretations, improved requirements, test ideas
decision-analyst
Structures trade-off analysis when real alternatives exist - only when a genuine choice must be made
Triggered by
real alternatives compare options prioritization conflict unclear ownership trade-offs
Output
Decision point, decision owner, alternatives, comparison table, recommendation or decision path

Key principles
  • No code. Works with requirements, scope, decisions, workflow, and acceptance criteria only.
  • Honesty markers. Marks assumptions as ASSUMPTION: and unresolved items as OPEN QUESTION:
  • Max 3 questions. Limits blocking questions to 3 per iteration. Continues with bounded assumptions when interaction is unavailable.
  • Behavior over implementation. Prefers requirement-level behavioral statements rather than prescribing solution design.
  • No duplication. Each specialist adds only what the requirements-analyst did not cover. Agents do not restate each other's findings.
  • Each agent runs once. Re-entry is blocked. If a later trigger targets a completed agent, the concern is folded into the merge step.
  • Context is passed forward. Scope, workflow, and integration findings are shared with subsequent agents so they do not re-derive earlier conclusions.
  • Conflict order. Compliance > Business outcome > Operational safety > Testability > Scope minimization. Remaining conflicts surface as DECISION POINT:
